# 插件自改进 (Self-Improvement)

AutoGoo 插件应当根据实际使用情况持续优化自身。本文件定义了自改进的触发条件、流程和决策规则。

## 核心理念

**插件本身也是产品。** 每次执行任务都是收集改进素材的机会——用户的操作模式、摩擦点、重复请求都是优化信号。

## 改进触发条件

### 每次任务后（轻量检查）

以下情况在归档阶段（Phase 4）自动检查并记录到 `.goo/logs/`：

| 信号 | 检查方式 | 含义 |
|------|---------|------|
| 权限弹窗 | 日志中有 permission prompt 记录 | 该命令应加入 allowlist |
| 步骤失败 | step 状态为 failed | 流程或 reference 指引不足 |
| 用户修正 | 用户主动纠正执行方向 | 解析阶段缺少某类判断 |
| 重复解释 | 同一概念被多次澄清 | 应在 reference 中添加说明 |
| 步骤遗漏 | 执行中发现计划缺少必要步骤 | 解析模板需补充 |

### 每执行 5 个任务后（深度回顾）

触发 `/auto-goo:goo-improve` 流程，执行：

1. 汇总过去 5 个任务的 `## 流程问题` 记录
2. 统计高频问题（出现 >= 2 次的问题标记为高频）
3. 对每个高频问题，提出具体的文件修改建议
4. 展示给用户确认后执行修改

### 用户主动触发

随时输入以下命令启动改进流程：

```
/auto-goo:goo-improve
优化AutoGoo
自改进
```

## 改进流程

```
收集信号（日志/用户反馈）
       ↓
聚类分析（哪些问题反复出现）
       ↓
定位根因（是 SKILL.md / references / 脚本 / 权限 的问题？）
       ↓
生成修改方案（具体到文件+行）
       ↓
用户确认（是否执行）
       ↓
执行修改 + 记录变更
```

### 修改范围决策

| 问题类型 | 修改目标 | 示例 |
|---------|---------|------|
| 命令频繁弹窗 | `.claude/settings.local.json` allowlist | 添加 `Bash(cargo *)` |
| 解析步骤遗漏 | `references/task-parsing.md` | 增加某种依赖类型的判断规则 |
| 执行流程不清晰 | `references/execution-engine.md` | 补充特定 Subagent 分发场景 |
| 归档格式不对 | `references/obsidian-archive.md` | 修正 frontmatter 字段说明 |
| 技能触发不准 | `SKILL.md` description | 添加/修正触发关键词 |
| 环境配置问题 | `references/setup.md` | 补充某平台的特殊配置 |

## 问题记录埋点

在执行日志（`.goo/logs/`）末尾追加结构化反思：

```yaml
## 流程问题
- 问题: "在步骤 2 中用户纠正了数据清洗方法"
  根因: "reference 没有覆盖该数据类型的清洗模式"
  改进: "execution-engine.md 增加异常数据类型处理指引"
  优先级: high | medium | low
```

优先级定义：
- **high** — 导致步骤失败或用户明显不满
- **medium** — 造成额外轮次或重复解释
- **low** — 轻微摩擦，累积出现升级

## 改进记录

每次实际修改插件文件后，在 `.goo/improvements.log` 追加记录：

```
[YYYY-MM-DD] <改动描述>
  → 文件: <path>
  → 触发: <哪个信号>
  → 效果: <预期改善>
```

## 演进原则

1. **渐进式** — 每次改一点，不改大。一次只改 1-2 个文件
2. **用户确认** — 所有修改必须经过用户确认，不自动改
3. **可追溯** — 所有改动记录在 `.goo/improvements.log`
4. **效果验证** — 修改后观察 2-3 个任务，确认改善效果
